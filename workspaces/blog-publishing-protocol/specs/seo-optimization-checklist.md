# SEO Optimization Checklist: 47-Point Verification

Run this checklist on every article before publishing. Each point is mandatory.

---

## SECTION 1: TITLE OPTIMIZATION (5 points)

- [ ] **Length**: 50-60 characters (Google displays 50-60 chars; longer gets cut off)
  - Check: Copy title to tool like titlecharacterlength.com
  
- [ ] **Primary keyword**: Included in title (ideally at start of title)
  - Example GOOD: "How to Implement AI in Existing Systems"
  - Example BAD: "A Complete Guide to System Updates and AI Features"
  
- [ ] **Format**: One of these patterns:
  - "[Keyword] Guide / How-To / Tutorial"
  - "How to [Keyword]"
  - "[Keyword] + [Modifier]: Complete Guide / 2025 Update"
  
- [ ] **Power words**: At least one of: "Complete", "Proven", "Ultimate", "Step-by-Step", "Best", "Real", "2025", "Essential"
  - Example: "The Complete Guide to AI Integration" (has "Complete")
  
- [ ] **Clickability**: Does it make reader want to click?
  - Self-test: Would you click this in Google results? If no, rewrite.

---

## SECTION 2: META DESCRIPTION (3 points)

- [ ] **Length**: 155-160 characters (test at snippetwriter.com or SERP simulator)
  
- [ ] **Content**: Includes keyword once + benefit/CTA
  - Example: "Learn how to integrate AI into existing systems. 7-step framework + real case study. Get started today."
  
- [ ] **CTA**: Ends with actionable language
  - "Learn more", "Read the guide", "Get started", "See how", "Discover", NOT just "..."

---

## SECTION 3: HEADING STRUCTURE (7 points)

- [ ] **H1 count**: Exactly ONE H1 per page (should match title)
  
- [ ] **H2 count**: 3-7 H2 sections (typical: 5 H2s for 2,000-word article)
  
- [ ] **H3 nesting**: Each H2 has 1-3 H3s; no skipped levels (H1 → H3 is wrong)
  
- [ ] **Heading keywords**: Primary keyword in H1; secondary keywords in 40-50% of H2/H3s
  - Example: If primary = "AI implementation", secondary = "integration steps", "business case"
  - Count keywords in headings; aim for 40-50% of headings to include keywords naturally
  
- [ ] **Heading length**: H2s ≤ 8 words; H3s ≤ 6 words (scannable)
  - Example BAD: "Understanding the Process of Implementing AI in Your Business Systems"
  - Example GOOD: "How to Implement AI"
  
- [ ] **Hierarchy check**: Outline headings; verify structure makes sense
  - H1 (title)
    - H2 (section)
      - H3 (subsection)
      - H3 (subsection)
    - H2 (section)
      - H3 (subsection)
  
- [ ] **Keyword spelling**: All keyword mentions in headings spelled identically (consistency)

---

## SECTION 4: BODY CONTENT (9 points)

- [ ] **Opening 100 words**: Includes primary keyword; answers "why care" immediately
  - Test: Read first 100 words alone. Does it answer the search intent? If no, rewrite.
  
- [ ] **Keyword density**: Primary keyword 0.5-1.5% of word count
  - Calculation: (keyword count ÷ total words) × 100 = percentage
  - For 2,000-word article: primary keyword appears 10-30 times
  - Check with SEMrush free keyword density tool or manual count
  
- [ ] **Secondary keywords**: Each secondary keyword 0.2-0.5% (appears 1-3 times naturally)
  
- [ ] **Keyword placement**: Keywords in first 100 words + H2s + body sections + conclusion
  - Don't keyword stuff (if it reads awkwardly, rewrite it)
  
- [ ] **Paragraph length**: Max 3-4 sentences per paragraph (scannable)
  - Count sentences in paragraphs; break any with 5+ sentences
  
- [ ] **Active voice**: 80%+ of sentences use active voice
  - Test: Read article; rewrite passive sentences to active
  - Passive: "The software was implemented by our team"
  - Active: "Our team implemented the software"
  
- [ ] **Sentence length**: Average ≤ 15 words per sentence
  - Count total words ÷ sentence count; aim for < 15
  
- [ ] **Readability score**: Flesch Reading Ease ≥ 60 (6th-grade reading level)
  - Test with Grammarly, Hemingway Editor, or SEMrush
  - Score > 60 = Easy to understand; < 40 = Too complex
  
- [ ] **Bolded keywords**: 2-3 bolded keywords per 500 words (visual emphasis without over-doing)
  - Should be keyword-rich phrases that deserve emphasis

---

## SECTION 5: INTERNAL LINKING (5 points)

- [ ] **Link count**: 5-7 contextual internal links per article
  
- [ ] **Pillar link**: At least 1 link to pillar page (usually in intro + conclusion)
  - For cluster articles: Must link back to pillar page
  - For pillar pages: Link to all cluster articles in TOC
  
- [ ] **Anchor text**: Keyword-rich, natural language (not "click here")
  - Good: "Our complete [keyword] guide covers..."
  - Bad: "Learn more" (generic) or "keyword keyword keyword" (over-optimized)
  
- [ ] **Link placement**: Links placed contextually within sentences, not at paragraph end
  - Good: "As discussed in our [guide to microservices architecture](#), you should..."
  - Bad: Separated: "Read more about microservices architecture [here](#)."
  
- [ ] **Link testing**: All links verified to work (no 404s)
  - Click each link in preview mode; confirm landing page is relevant and works

---

## SECTION 6: CONTENT FORMAT & SCANNABILITY (6 points)

- [ ] **Bulleted lists**: Use for 3+ items (improves scannability)
  - 5+ items should always be bulleted/numbered
  
- [ ] **Numbered lists**: Use for step-by-step processes ("How to..." articles)
  
- [ ] **Tables**: Use for comparisons or complex data
  - At least 1 table per 2,000 words (if data exists)
  
- [ ] **Bolded text**: Key phrases bolded for visual emphasis (not every other word)
  
- [ ] **Short sections**: Every section ≤ 300 words before a visual break (image, heading, list)
  - Prevents "wall of text" feeling
  
- [ ] **White space**: Sufficient spacing between sections; not cramped
  - Preview on phone; should feel readable, not overwhelming

---

## SECTION 7: IMAGES & MEDIA (6 points)

- [ ] **Image count**: Minimum 1 image per 500 words
  - For 2,000-word article: minimum 4 images
  
- [ ] **Image optimization**: Each image ≤ 200KB
  - Compress with TinyPNG, Squoosh, or Cloudinary
  
- [ ] **Image format**: WebP preferred (better compression); JPEG/PNG acceptable
  - Test: Load article on mobile; images should load fast
  
- [ ] **Alt text**: Every image has alt text (8-12 words; includes keyword once naturally)
  - Example: `alt="custom software development architecture diagram showing microservices pattern"`
  - Not: `alt="image123"` or `alt="AI AI AI picture"`
  
- [ ] **Image captions**: 1-2 sentence caption explaining relevance
  - Example caption: "Figure 1: A microservices architecture allows independent scaling of AI components, reducing overall system latency by 40%."
  
- [ ] **Visual variety**: Mix of screenshots, diagrams, charts, graphs (not all same type)
  - At least one of: diagram/flowchart, infographic, screenshot, chart/graph, table

---

## SECTION 8: METADATA & TECHNICAL (6 points)

- [ ] **Canonical URL**: Set to self-referential (prevent duplicate content issues)
  - Every page's canonical should point to itself: `<link rel="canonical" href="https://yoursite.com/blog/article-slug/">`
  
- [ ] **URL slug**: Keyword-rich, lowercase, hyphens only (no underscores, numbers, or special chars)
  - Good: `/blog/how-to-implement-ai-in-existing-systems/`
  - Bad: `/blog/article_123/` or `/blog/how%20to%20implement%20AI/`
  
- [ ] **Open Graph tags**: Filled in for social sharing
  - `og:title`, `og:description`, `og:image`, `og:url` all present
  
- [ ] **Twitter Card**: Set to `summary_large_image` (shows image on Twitter)
  - `<meta name="twitter:card" content="summary_large_image">`
  
- [ ] **Schema markup**: BlogPosting JSON-LD added (or FAQPage if article has FAQ)
  - Test with Google Structured Data Testing Tool; should show 0 errors
  
- [ ] **Author & date**: Author name and publish date included in article (visible to reader)
  - User should see: "By [Author Name] | Published May 2, 2025 | Updated [date if updated]"

---

## SECTION 9: READABILITY & UX (4 points)

- [ ] **Grammar & spelling**: No errors (run through Grammarly before submitting)
  
- [ ] **Tone**: Consistent throughout; matches brand voice
  - If article starts professional, maintain professional; don't switch to casual
  
- [ ] **Mobile preview**: Article readable on phone (responsive layout, images load, text not cramped)
  - Physically test on phone or use browser DevTools mobile preview
  
- [ ] **CTA placement**: Clear call-to-action at end (link to pillar page or product)
  - Example: "Ready to implement AI? Read our complete [AI Integration Playbook](#) for step-by-step guidance."

---

## SECTION 10: COMPETITOR COMPARISON (3 points)

- [ ] **Word count**: Matches or exceeds top-3 ranking articles by 10-20%
  - If top-3 average 2,200 words, aim for 2,400-2,600 words
  
- [ ] **Content depth**: Covers all major H2 topics competitors cover, PLUS unique sections
  - If every competitor covers "Implementation Steps", include that
  - Add unique section competitors don't cover (e.g., "Hidden Costs to Avoid" or "ROI Calculator")
  
- [ ] **Examples & data**: Includes same number or more examples/case studies/statistics as top-3
  - If competitors have 3 case studies, include 3-4

---

## SECTION 11: FINAL CHECKS (2 points)

- [ ] **Plagiarism test**: Article is unique (run through Copyscape or Grammarly plagiarism check)
  - Should show 0% plagiarism
  
- [ ] **Broken links test**: All internal and external links work
  - Use link checker tool or manually click each link in preview mode

---

## AUTOMATION OPPORTUNITIES

**These can be automated via CI/CD pipeline:**

1. **Word count validation** → Script checks word count is within range
2. **Keyword density checking** → Script counts keyword occurrences
3. **Readability scoring** → Integrate Flesch Reading Ease API
4. **Link validation** → Automated broken link checker
5. **Image optimization** → Automated image compression on upload
6. **Alt text validation** → Script checks all images have alt text
7. **Heading structure validation** → Script verifies H1/H2/H3 hierarchy
8. **Mobile rendering test** → Automated Lighthouse/PageSpeed Insights test
9. **Schema markup validation** → Automated structured data testing
10. **Grammar checking** → Integrate Grammarly API

## Common Failures & Fixes

| Failure | Why It Happens | How to Fix |
|---|---|---|
| Low CTR (high impressions, few clicks) | Title not compelling; meta description generic | Add power word to title; make meta description more specific/benefit-focused |
| Article not ranking (no impressions after 2 weeks) | No internal links from pillar; weak content | Get backlink to pillar page; update internal links; expand word count 20% |
| Readers bouncing (high bounce rate) | Article doesn't match search intent | Check article topic matches keyword intent; rewrite opening to answer immediately |
| Low readability score | Sentences too long; paragraphs too dense | Break sentences < 15 words; max 3-4 sentences per paragraph; add visual breaks |
| Schema validation errors | JSON-LD syntax wrong | Validate with Google Structured Data Testing Tool; fix errors; retest |
| Images look blurry on mobile | Image too small or low resolution | Use higher resolution image (1200x800px minimum for hero); optimize after resizing |
| Links don't match anchor text | Anchor text says "AI integration guide" but links to unrelated article | Fix links to match anchor text meaning; use relevance principle |
| URL is too long | Original slug auto-generated | Shorten to essential keywords only; max 50-60 characters |
